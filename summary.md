# SeeMusic 项目进度总结

更新时间：2026-05-07

本总结基于当前工作区实际代码状态整理，包含尚未提交到 Git 的本地改动。

## 一句话结论

SeeMusic 现在已经完成了“可继续开发的工程骨架 + 一条可演示的音频分析 MVP 链路”，但距离“完整可交付的智能扒谱 / 演唱评估 / 社区产品”还有明显差距。

- 如果按工程骨架看，完成度大约在 60% 到 70%。
- 如果按最终产品能力看，完成度更接近 30% 到 40%。

## 已经完成的部分

### 1. 后端基础设施基本成型

- 已有 `ASP.NET Core 8 Web API` 后端。
- 已接入 `EF Core + MySQL + JWT + Swagger`。
- 已有 `docker-compose.yml`，可以拉起 `MySQL + backend`。
- 后端项目和测试项目都能编译。

当前已经落地的主要接口包括：

- `POST /api/v1/auth/register`
- `POST /api/v1/auth/login`
- `POST /api/v1/auth/refresh`
- `POST /api/v1/auth/logout`
- `GET /api/v1/users/me`
- `PUT /api/v1/users/me`
- `POST /api/v1/users/me/avatar`
- `POST /api/v1/media/upload`
- `POST /api/v1/transcriptions/analyze`

### 2. 用户与媒体的基础能力已经有后端实现

- 注册、登录、刷新 token 已实现。
- 用户资料读取和更新已实现。
- 媒体上传已实现，文件会保存到 `backend/uploads`，同时写入数据库。
- 为了本地演示流程，上传接口支持匿名上传，并会自动兜底到一个 demo 用户。

### 3. 节拍分析能力已经是真实代码，不只是文档设想

当前工作区新增了完整的 `BeatAnalysisService`，已经能对音频做这些分析：

- BPM 估计
- 拍点时间序列提取
- 节奏稳定度计算
- 置信度计算
- 简单拍号推测

这部分不是纯占位返回，而是有实际分析逻辑和测试覆盖的功能。

### 4. 前端已经有完整的多窗口原型

桌面端是 `WPF (.NET Framework 4.8)`，主要页面都已经搭出来了：

- 首页
- 登录 / 注册
- 扒谱页
- 演唱评估页
- 社区页
- 个人中心页

页面完成度从“视觉原型”角度看并不低，导航和基础交互也已经串起来了。

### 5. 前后端已经打通了一条 MVP 链路

`TranscriptionWindow` 和 `SingingEvaluationWindow` 都已经接入后端分析接口，能跑通这条流程：

1. 选择本地音频
2. 上传到后端
3. 调用分析接口
4. 把节拍分析结果回填到界面

这说明项目已经不是纯静态原型，而是进入了“局部可运行”的阶段。

### 6. 后端测试已经能跑通

我在 2026-05-07 实际执行了：

```bash
dotnet test backend.Tests/SeeMusic.Backend.Tests.csproj
```

结果是 `6/6` 通过。

目前已有测试主要覆盖：

- JWT token 生成与校验
- 节拍分析服务

## 现在还没完成的部分

### 1. “智能扒谱”还没有真正落到乐谱生成

虽然现在有 `Transcription` 接口和扒谱界面，但现阶段真正做成的核心是“节拍分析”，不是完整扒谱。

还没看到这些关键能力真正落地：

- 从音频生成真实音符数据
- 乐谱持久化存储
- 音轨 / 音符级编辑数据模型
- 导出 PDF / MIDI / PNG / MusicXML
- 重新分析后的版本管理

现在的 `ScoreId` 更像流程占位，不是完整乐谱产物。

### 2. 扒谱选项还没有形成真实算法分支

前端和请求模型里已经有：

- `SeparateMelody`
- `SeparateAccompaniment`

但当前后端分析逻辑并没有真正使用它们去驱动旋律分离或伴奏分离算法，所以这部分还属于“接口先行，能力未落地”。

### 3. 演唱评估仍然是“节奏评估 MVP”

现在演唱评估页已经能根据节拍分析结果给出分数和建议，但它还不是完整的 AI 演唱评估系统。

尚未落地的能力包括：

- 音准检测
- 音高曲线分析
- 情感 / 表达分析
- 咬字 / 发声 / 句法维度评分
- 参考作品对齐评分

所以更准确地说，目前它是“节奏分析驱动的演示版评估页”。

### 4. 登录注册页仍然是前端模拟流程

后端鉴权 API 已经写了，但 `LoginWindow` 里目前还是本地 `MessageBox` 模拟登录 / 注册成功，没有真正调用后端。

也就是说：

- 后端 auth 已有
- 前端 auth UI 已有
- 但两者还没有真正接上

### 5. 社区和个人中心主要还是原型态

从当前代码看：

- 社区页主要是静态卡片 + 详情联动
- 上传乐谱还是提示弹窗
- 个人中心的缓存清理是模拟逻辑
- 头像上传接口后端也只是返回占位结果

这些模块说明“页面方向已经明确”，但距离可用产品还差后端模型、接口和真实交互闭环。

### 6. 文档存在过时和不同步问题

当前仓库里有几类文档明显没有完全同步到最新代码状态：

- 根目录 `README.md` 里仍有“基础框架”表述
- 文档中有 `client/` 与实际 `frontend/` 目录混用
- `frontend/README.md` 还是模板内容
- `api.md`、`database.md` 有不少是设计稿，不完全等于当前实现

这意味着“文档愿景”明显超前于“当前代码落地”。

## 我对当前阶段的判断

### 更像什么

现在的 SeeMusic 更像：

- 一个已经搭好前后端骨架的音乐产品原型
- 一个开始验证“音频分析 -> UI 展示”闭环的 MVP
- 一个适合继续冲刺核心功能的开发中项目

### 还不像什么

它现在还不像：

- 一个已经完成核心 AI 能力的成品
- 一个用户可以直接稳定使用的桌面应用
- 一个社区、个人中心、扒谱、评估都已经联动完毕的正式版本

## 建议你把它理解为哪个阶段

我会把这个项目定义在：

**“基础架构已完成，核心能力正在从原型过渡到 MVP，可演示但还未产品化。”**

如果要更直白一点：

- 不是从 0 开始，已经做了不少。
- 也远没到“基本完成”。
- 当前最扎实的成果是：后端框架、基础鉴权、媒体上传、节拍分析、两页前端联调。
- 当前最大的缺口是：真正的扒谱、完整演唱评估、真实登录流程、社区产品化。

## 最值得优先推进的下一步

如果接下来要继续做，我建议优先级是：

1. 把登录 / 注册前端接到真实 auth API，先打通完整用户链路。
2. 把 `transcriptions/analyze` 从“节拍分析”扩展成真正的乐谱结果模型。
3. 明确演唱评估的 MVP 范围，先决定是只做节奏评分，还是继续补音准能力。
4. 给 `auth`、`media`、`transcription` 增加更多集成测试。
5. 统一修正文档，把“设计稿”和“当前实现”分开说明。

## 最后结论

截至 2026-05-07，SeeMusic 已经完成到“工程框架稳了，局部功能跑通了，但核心产品能力还在中段开发”的程度。

如果你要一句最简短的判断：

**这是一个已经脱离纯原型、但还没有进入成品阶段的项目。**
