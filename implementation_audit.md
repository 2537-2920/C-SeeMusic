# SeeMusic 配置与功能缺口排查

更新时间：2026-05-09

本文件基于当前仓库代码状态整理，包含当前工作区中尚未提交的本地改动。

## 1. 数据库连接逻辑检查

### 1.1 结论

- 现在没有 `.env` 文件，不代表数据库连接逻辑有问题。
- 这个项目当前主要依赖 `ASP.NET Core` 默认配置链路，而不是 `.env` 自动加载。
- 你的云服务器是 Windows 也没有关系，只要后端进程能读到环境变量，或者能读到生产配置文件，就可以正常连 MySQL。

换句话说：

- 当前项目是“支持环境变量覆盖”的。
- 但它不是“必须依赖 `.env` 文件”的。
- 如果你原本期待像 Node 项目那样自动读 `.env`，那这套逻辑目前没有做。

### 1.2 当前实际配置来源

从代码看，后端现在的配置来源是这几层：

1. `backend/appsettings.json`
2. `backend/appsettings.Development.json`
3. 运行时环境变量

数据库和 JWT 的读取方式是：

- `ConnectionStrings:DefaultConnection`
- `Jwt:Secret`
- `Jwt:Issuer`
- `Jwt:Audience`
- `Jwt:ExpiresInMinutes`

Docker Compose 里也已经在直接注入环境变量：

- `ConnectionStrings__DefaultConnection`
- `Jwt__Secret`
- `Jwt__Issuer`
- `Jwt__Audience`

这里的双下划线 `__` 是 ASP.NET Core 约定，用来映射配置层级。

### 1.3 这套逻辑对 Windows 服务器是否可用

可用。

如果你把后端直接部署到 Windows 云服务器，而不是跑 Linux 容器，常见做法有两种：

1. 在 Windows 上配置系统环境变量
2. 在服务器本地放 `appsettings.Production.json`

这两种都不需要 `.env`。

如果你要用 Windows 系统环境变量，变量名可以直接写成：

```powershell
ConnectionStrings__DefaultConnection=Server=127.0.0.1;Port=3306;Database=seemusic;Uid=xxx;Pwd=xxx;
Jwt__Secret=your-production-secret-at-least-32-characters
Jwt__Issuer=SeeMusic
Jwt__Audience=SeeMusic
Jwt__ExpiresInMinutes=120
```

例如用 PowerShell 持久写入机器级环境变量：

```powershell
[System.Environment]::SetEnvironmentVariable(
  "ConnectionStrings__DefaultConnection",
  "Server=127.0.0.1;Port=3306;Database=seemusic;Uid=xxx;Pwd=xxx;",
  "Machine"
)

[System.Environment]::SetEnvironmentVariable(
  "Jwt__Secret",
  "your-production-secret-at-least-32-characters",
  "Machine"
)
```

设置完成后，重启应用进程或服务即可。

### 1.4 我看到的实际问题和风险

下面这些是当前仓库里真正值得注意的点。

#### 1. 项目并不会自动读取 `.env`

虽然 `.gitignore` 已经忽略了：

- `.env`
- `.env.*`
- `appsettings.Local.json`
- `appsettings.*.Local.json`

但代码里没有看到任何 `.env` 解析逻辑，也没有 `dotenv` 类依赖。

这意味着：

- 你可以自己创建 `.env`
- 但当前后端不会自动读取它
- 除非你通过 Docker Compose 的 `env_file` 或变量插值显式接进来

#### 2. 当前提交的 `appsettings*.json` 只是占位值

`backend/appsettings.json` 和 `backend/appsettings.Development.json` 里的数据库账号、密码、JWT Secret 仍然是 `CHANGE_ME` 占位符。

这意味着：

- 如果服务器没有正确配置环境变量
- 或者没有额外的生产配置文件

应用启动时就会失败，或者连不上数据库。

#### 3. 没有看到现成的 `appsettings.Production.json`

当前仓库里没有看到生产环境专用配置文件模板。

所以部署时容易出现两个问题：

1. 运维同学不知道该把生产连接串写到哪里
2. 大家误以为 `.env` 会自动生效

#### 4. Docker Compose 里把生产敏感信息直接写死了

`docker-compose.yml` 里直接内联了：

- 数据库连接串
- JWT Secret

这适合本地联调，不适合正式部署。

更稳妥的方式通常是：

1. 用 `.env` 只做 Docker Compose 变量插值
2. 或者改成 `env_file`
3. 或者直接用服务器环境变量 / CI Secret 注入

#### 5. 启动时强依赖数据库立刻可连通

当前启动流程里有两处会在应用启动早期依赖数据库：

1. `ServerVersion.AutoDetect(connectionString)`
2. `context.Database.Migrate()`

这意味着：

- 数据库地址错误，应用会直接启动失败
- 数据库还没准备好，应用也可能启动失败
- 数据库用户权限不够，迁移也会失败

在 Docker Compose 里你们已经靠 MySQL healthcheck 做了部分兜底，但在 Windows 云服务器直部署时，这个问题需要你自己保证。

#### 6. 仓库里没看到 EF Core Migration 文件

当前我没有在仓库里看到 `backend/Migrations/` 之类的迁移文件。

这会带来一个不确定点：

- 代码里调用了 `Database.Migrate()`
- 但仓库里看不到可执行的迁移历史

所以在一台全新的服务器上：

- 如果没有未提交的迁移文件，自动建表流程大概率是不完整的
- 至少目前不能从仓库里确认“新库一定能自动初始化成功”

这个点建议尽快补齐。

#### 7. 数据库类型当前是硬编码为 MySQL

后端现在固定使用：

- `Pomelo.EntityFrameworkCore.MySql`
- `UseMySql(...)`

这本身没问题，但要注意：

- 你的服务器是 Windows 不重要
- 真正重要的是数据库是不是 MySQL

如果你服务器上实际准备接的是 SQL Server，而不是 MySQL，那当前代码不能直接复用。

#### 8. 前端默认后端地址和后端默认端口不一致

这不是数据库问题，但它会影响你联调时对“是不是数据库没连上”的判断。

当前看到的配置是：

- 前端 `frontend/App.config` 默认访问 `http://localhost:5000`
- Docker Compose 后端暴露的是 `5001`
- README / 文档里也多次写的是 `5001`

所以如果前端请求打不通，未必是数据库问题，也可能只是端口不一致。

### 1.5 我对数据库配置的建议

如果你们准备部署到 Windows 云服务器，我建议这样做：

1. 保留现在的 `appsettings.json` 作为占位模板，不放真实生产密码。
2. 在服务器上使用环境变量覆盖 `ConnectionStrings__DefaultConnection` 和 `Jwt__Secret`。
3. 单独补一个不入库的 `appsettings.Production.json` 模板说明文档，避免团队误会 `.env` 会自动生效。
4. 补齐 EF Core Migration 文件，确保新库可以稳定初始化。
5. 把前端默认后端地址从 `5000` 统一到你们实际部署端口。
6. 如果还要继续用 Docker Compose，再决定要不要补一个 `.env.example`，但要明确它是给 Compose 用的，不是给 ASP.NET Core 自动读的。

### 1.6 最简短判断

最简短地说：

- 当前数据库连接逻辑不是“缺 `.env` 所以有问题”
- 而是“配置来源没问题，但生产部署约定还不够清晰”

真正要补的是：

- 生产配置规范
- 迁移文件
- 端口与环境说明

## 3. 如果只按优先级看，最该先补什么

如果你想先把项目做成“能跑、能演示、能部署”的版本，我建议优先顺序是：

1. 补齐生产环境配置约定，明确 Windows 服务器到底用环境变量还是 `appsettings.Production.json`。
2. 补 EF Core Migration，确保新数据库可以稳定初始化。
3. 把前端登录 / 注册真正接到后端 auth API。
4. 统一前后端端口和环境配置，避免联调误判。
5. 把 `transcriptions/analyze` 从“节拍分析结果”扩展到“真正乐谱结果模型”。
6. 再决定演唱评估是先做“节奏版 MVP”，还是继续补音准检测。

## 4. 最后判断

数据库配置这块的核心问题，不是“少了 `.env` 文件”，而是“部署规范还没收口”。

功能实现这块的核心问题，不是“完全没做”，而是“已经有前后端骨架和局部闭环，但大部分产品能力还停在原型或半成品阶段”。
